The user wants me to provide the payment-related API documentation in text format for the frontend developer. Let me review what was implemented earlier - the `POST /oms/order/pay` endpoint with `PayOrderDTO` and `PayOrderVO`.

Let me recall the details:

**PayOrderDTO:**
- `id` (Long, required) - 订单id
- `paymentType` (Integer, optional) - 支付方式, 0=银联，1=微信，2=支付宝

**PayOrderVO:**
- `id` (Long) - 订单id
- `sn` (String) - 订单编号
- `paymentType` (Integer) - 支付方式
- `payAmount` (BigDecimal) - 支付金额
- `gmtPay` (LocalDateTime) - 支付时间
- `state` (Integer) - 订单状态

Also the `POST /oms/order/update/state` was fixed to auto-set gmtPay when state=3.

Let me write this out clearly.

**支付订单接口**

**接口地址**：`POST /oms/order/pay`

**权限**：需登录（user 角色）

**请求参数**（JSON Body）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | Long | 是 | 订单id |
| `paymentType` | Integer | 否 | 支付方式：0=银联，1=微信，2=支付宝。不传则使用下单时选择的支付方式，默认为0 |

请求示例：
```json
{
  "id": 2000,
  "paymentType": 0
}
```

**返回值**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long | 订单id |
| `sn` | String | 订单编号 |
| `paymentType` | Integer | 支付方式：0=银联，1=微信，2=支付宝 |
| `payAmount` | BigDecimal | 实际支付金额 |
| `gmtPay` | String | 支付时间，格式 `yyyy-MM-dd HH:mm:ss` |
| `state` | Integer | 订单状态，支付成功后为 3 |

返回示例：
```json
{
  "state": 200,
  "message": "OK",
  "data": {
    "id": 2000,
    "sn": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "paymentType": 0,
    "payAmount": 5100.00,
    "gmtPay": "2026-05-08 15:30:00",
    "state": 3
  }
}
```

**错误情况**：

| 场景 | 状态码 | 提示信息 |
|------|--------|---------|
| 订单不存在 | 400 | 订单不存在 |
| 订单状态不是未支付 | 400 | 当前订单状态不支持支付 |
| 操作他人订单 | 403 | 无权操作此订单 |

**使用说明**：

1. 用户下单后（`POST /oms/order/add`），订单状态为 0（未支付），此时可调用支付接口
2. `paymentType` 可以不传，后端会自动使用下单时选择的支付方式
3. 当前版本为模拟支付，点击即完成支付，后续接入微信/支付宝后前端无需改接口，只需在支付页面对接对应的支付SDK即可
4. 原有的 `POST /oms/order/update/state` 接口仍然可用，当 state 传 3 时也会自动设置支付时间，但建议支付流程统一使用 `/pay` 接口
